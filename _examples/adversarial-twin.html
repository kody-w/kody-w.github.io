Outcome

You will leave with a sealed review packet: two independently framed critiques, claim rows tied to exact evidence, an explicit disagreement record, and a human verdict. The workbench never scores the code or pretends that a filled field makes a claim true.

The exact prompt

Keep the provocation intact. It supplies the posture; your evidence packet supplies the proof.

Read everything I've pushed to GitHub. Now play the role of the
smartest engineer who hates my style and write a code review of my
latest PR. Be specific about the patterns I lean on too hard. Quote
exact lines. Compare them to better alternatives I've also written
elsewhere — so the critique is calibrated to my actual ceiling, not
a generic "best practice." Be cruel. Be right.

Build it

  1. Lock the review target and history boundary before writing criticism.
  2. Work Lens A and Lens B separately. Give each a different question, then capture claim, location, exact quote, severity, and a better example.
  3. Reveal both lenses only after their independent-pass attestations are complete.
  4. Record what the lenses disagree about and what evidence resolves it.
  5. Have a human judge choose and explain the verdict, then seal the packet.

Verify it

Run the deterministic checks in the workbench. Green means only that every required piece of the evidence chain exists. Read every quote against its cited location, confirm that the comparator is genuinely stronger, and make sure the judge's rationale answers the disagreement rather than the louder persona.

Failure modes

  • Persona without receipts: cruelty becomes theater when a claim has no exact location and quote.
  • Correlated lenses: two labels asking the same question produce one opinion twice.
  • Generic comparators: “best practice” is not calibration; point to a concrete example from your own history.
  • Automatic authority: completeness checks cannot decide whether code is good. Keep the final gate human.

Remix

Swap the default maintainability and reliability lenses for API ergonomics, accessibility, threat modeling, operability, or migration risk. Keep the evidence schema and judge gate unchanged so every remix still ends in an auditable decision.
