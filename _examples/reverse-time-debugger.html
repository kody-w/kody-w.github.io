Outcome

You will produce a plain-text Markdown investigation packet containing the supplied stack trace, a learner-ordered evidence timeline, explicit unknowns and hypotheses, and notes from the user, on-call, and original-author perspectives. The workbench formats only what you enter; it performs no AI analysis, inference, diagnosis, or repair.

Lesson 2 of Prompt to Proof · 30 minutes. Bring a sanitized trace and observations, or use the fictional sample.

Build it

  1. Anchor the incident. Paste the trace exactly as observed. Do not turn an exception type into a cause before the surrounding evidence supports it.
  2. Load atomic observations. Put one fact per line, with a relative time such as T-42s when you have one. The timeline accepts 200 items and leaves excess lines in the intake instead of silently dropping them.
  3. Order the evidence. Move cards earlier or later with buttons or Alt + arrow keys. Then explicitly mark the order reviewed.
  4. Separate fact from inference. Write unknowns, describe possible data in flight, and label the developer-intent field as a hypothesis.
  5. Add human perspectives. Write evidence-bounded notes for the user, on-call engineer, and original author; do not manufacture quotations.

Verify it

The readiness indicators use visible rules only: a non-empty trace, at least three evidence items, a checked order review, completed inference-boundary fields, and all three perspective notes. The packet preview is generated deterministically from those fields and always includes a manual-only disclosure.

Read the exported timeline from oldest event to crash. For every transition, ask which supplied observation supports it. If no evidence does, move the statement to known unknowns or label it as a hypothesis.

Failure modes

  • Fixing instead of reconstructing. A plausible patch can end inquiry before the timeline is understood.
  • Turning stack frames into user behavior. A trace records execution, not the user's intent. Add intent only when observations support it.
  • Presenting author intent as fact. Keep the probable assumption in the hypothesis field and verify it with code history or the author later.
  • Writing synthetic voices. Perspective notes should summarize impact and constraints, not pretend to be direct testimony.
  • Leaking sensitive production data. Sanitize identifiers before pasting, saving locally, or exporting.

Remix

Add lanes for telemetry, user actions, deploy events, and human decisions, or run the exercise with two investigators who must reconcile their timelines using only shared evidence. The strongest remix is a packet that becomes more conservative as uncertainty rises—not one that sounds more certain.
